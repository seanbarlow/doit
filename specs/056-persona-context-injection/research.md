# Research: Project-Level Personas with Context Injection

**Feature Branch**: `056-persona-context-injection`
**Date**: 2026-03-26

## R-001: Persona Source Priority Placement

**Decision**: Priority 3 â€” between `roadmap` (priority 3 currently) and `current_spec` (priority 4). Shift roadmap to priority 4, current_spec to 5, related_specs to 6.

**Rationale**: Personas provide foundational user context that informs how specs and plans are interpreted. They should load before feature-specific content but after governance (constitution, tech-stack). The existing `completed_roadmap` source was added as a code-only default (not in YAML template) at priority 4 â€” it should move to priority 7 as reference material.

**Alternatives considered**:
- Priority 6 (after specs): Too late â€” personas inform spec interpretation
- Priority 1 (before constitution): Constitution governs all decisions; personas are secondary context

**Final priority order**:

| Priority | Source |
|----------|--------|
| 1 | constitution |
| 2 | tech_stack |
| 3 | personas |
| 4 | roadmap |
| 5 | current_spec |
| 6 | related_specs |
| 7 | completed_roadmap |

## R-002: Persona Generation Trigger

**Decision**: Generate `.doit/memory/personas.md` as a step in `/doit.roadmapit` when creating or updating the roadmap.

**Rationale**: The roadmap command already loads the constitution and understands project scope. Deriving personas from constitution stakeholder types and roadmap user-facing features is a natural extension. The `personas-output-template.md` from spec 053 provides the structure.

**Alternatives considered**:
- Standalone `/doit.personas` command: Adds workflow complexity; personas are a byproduct of understanding the project, not a standalone activity
- During `/doit.constitution`: Too early â€” constitution defines principles, not users
- During `/doit.researchit`: This generates *feature-level* personas, not project-level

## R-003: Project-Level vs Feature-Level Persona Precedence

**Decision**: When both `.doit/memory/personas.md` (project-level) and `specs/{feature}/personas.md` (feature-level) exist, feature-level personas take precedence for that feature's context.

**Rationale**: Feature-level personas are more specific and may include personas not relevant project-wide. The context loader already handles per-feature source loading for `current_spec` â€” this follows the same pattern.

**Implementation approach**: In the context loader, when loading for a command that has a current feature (detected via git branch), check for feature-level personas first. If found, use those instead of project-level. This is a load-time decision, not a merge.

## R-004: Context Loading Without Truncation

**Decision**: Personas are always loaded in full. No truncation is applied.

**Rationale**: User requirement â€” context sources should never be truncated. If total context budget is exceeded, the solution is to disable sources via per-command overrides, not to truncate content.

**Impact**: Persona files from the `personas-output-template.md` typically contain 2-4 personas at ~200-400 tokens each, totaling 800-1600 tokens. This is well within typical context budgets.

## R-006: `doit init` Behavior

**Decision**: Do NOT add `personas.md` to `MEMORY_TEMPLATES` in template_manager.py. The file is generated by `/doit.roadmapit`, not scaffolded by `doit init`.

**Rationale**: Unlike constitution.md and roadmap.md which are templates to be filled, personas.md is a derived artifact. An empty persona template would provide no value and might confuse users into manually editing it.

**Alternative considered**: Adding a placeholder â€” rejected because it would conflict with the generated output from roadmapit.

---

## R-007: Command Override Defaults

**Decision**: Enable personas for `researchit`, `specit`, and `planit`. Disable for `constitution`, `roadmapit`, `taskit`, `implementit`, `testit`, `reviewit`, and `checkin`.

**Rationale**: Only the three target commands benefit from persona context. Loading personas for implementation/testing commands adds noise without value.

**Implementation**: Add command overrides in `CommandOverride.default_commands()` to disable personas for non-target commands.
